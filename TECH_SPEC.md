# ТЗ: local-document-indexer

## 1. Цель проекта

Реализовать локальный пайплайн индексации документов на Kotlin, который:

- загружает документы из файловой системы;
- извлекает из них текст;
- разбивает текст на чанки двумя разными стратегиями;
- генерирует эмбеддинги через локальный Ollama;
- сохраняет индекс и метаданные локально;
- позволяет выполнять простой semantic search;
- позволяет сравнить результаты двух стратегий chunking.

Проект делается как учебный, поэтому архитектура должна быть понятной, модульной и пригодной для постепенного развития.

## 2. Архитектурный принцип проекта

Первая версия проекта строится по принципу `SQLite-first`.

Это означает:

- SQLite используется как основное локальное хранилище документов, чанков, метаданных и эмбеддингов;
- поиск в первой версии выполняется в приложении через полный перебор чанков и расчёт cosine similarity в Kotlin;
- структура проекта должна быть подготовлена к дальнейшему подключению FAISS как отдельного слоя векторного поиска.

Во второй итерации SQLite не заменяется полностью, а дополняется:

- SQLite остаётся источником текстов, документов и metadata;
- FAISS в будущем используется как ускоренный индекс для nearest-neighbor поиска по embedding-векторам;
- поиск может быть организован по схеме:
  - запрос -> embedding;
  - FAISS возвращает `topK chunkId`;
  - подробности чанков подтягиваются из SQLite.

## 3. Требования задания

Необходимо поддержать:

- источники: README, markdown, txt, pdf, исходный код;
- суммарный объём корпуса: минимум 20–30 страниц текста или эквивалент в коде;
- две стратегии chunking:
  - fixed-size;
  - structured;
- metadata для каждого чанка:
  - `source`;
  - `title` или `file`;
  - `section`;
  - `chunkId`;
- локальное сохранение индекса.

Основной путь реализации:

- SQLite как базовое хранилище;
- JSON как необязательный будущий экспортный формат для отладки и демонстрации;
- FAISS не обязателен в первой версии, но архитектурно должен быть предусмотрен как следующий этап развития.

## 4. Границы первой версии

## 6. Основной пайплайн индексации

Пайплайн должен быть реализован как последовательность шагов:

1. Сканирование директории и поиск поддерживаемых документов.
2. Извлечение текста из каждого документа.
3. Нормализация текста.
4. Разбиение текста на чанки.
5. Генерация metadata для каждого чанка.
6. Получение embedding-вектора для каждого чанка через Ollama.
7. Сохранение документов, чанков, metadata и embeddings в локальный индекс.

Индекс в контексте проекта — это локально сохранённая структура данных, содержащая:

- документы;
- чанки;
- метаданные;
- embedding-векторы.

## 7. Будущее развитие: поисковый пайплайн

Поиск не входит в MVP, но архитектура должна позволять добавить его в следующих итерациях.

Будущий поисковый сценарий может работать так:

1. Пользователь вводит текстовый запрос.
2. Приложение получает embedding запроса через Ollama.
3. Из локального индекса загружаются embeddings чанков выбранной стратегии.
4. Для каждого чанка считается cosine similarity между embedding запроса и embedding чанка.
5. Результаты сортируются по убыванию score.
6. Пользователю возвращается `topK` наиболее релевантных чанков.

Важно:

- в первой версии SQLite используется как хранилище, а не как нативный vector search engine;
- вычисление similarity в будущей реализации может выполняться в Kotlin-коде;
- архитектура должна позволять в будущем заменить brute-force поиск на FAISS-поиск без переписывания моделей данных.

## 8. Источники и загрузка документов

Система должна:

- рекурсивно обходить указанную директорию;
- находить поддерживаемые файлы;
- определять тип источника;
- собирать документы в унифицированный формат `RawDocument`.

Поддерживаемые расширения:

- `.md`;
- `.txt`;
- `.pdf`;
- `.kt`, `.java`, `.py`, `.js`, `.ts`, `.json`, `.yaml`, `.yml`;
- при желании `.html`.

Дополнение:

- README рассматривается как markdown-документ;
- статьи могут быть представлены файлами `md`, `txt`, `pdf` или `html`.

Результат:

- список документов в унифицированном формате `RawDocument`.

## 9. Извлечение текста

Для каждого документа система должна:

- извлечь текстовое содержимое;
- нормализовать переносы строк;
- удалить явно мусорные пустые блоки;
- сохранить базовые атрибуты документа.

Особенности:

- PDF извлекается через PDFBox;
- код читается как обычный текст;
- markdown/readme сохраняются как plain text без удаления заголовков;
- extractor не должен ронять весь пайплайн из-за одного проблемного файла.

## 10. Chunking

Нужно реализовать две стратегии.

### 10.1. FixedSizeChunker

Требования:

- режет текст по фиксированному размеру;
- поддерживает overlap;
- работает для всех типов документов.

Базовые параметры:

- `chunkSize = 1200` символов;
- `overlap = 200` символов.

### 10.2. StructuredChunker

Требования:

- разбивает по структуре документа;
- должен использовать максимально естественные смысловые границы.

Правила:

- markdown/readme: по `#`, `##`, `###`;
- code: минимум по файлам, желательно дополнительно по `class`, `fun`, `interface`, `object`, `enum`;
- txt/pdf: по крупным абзацам, разделителям или найденным заголовкам;
- если структура не найдена, допускается fallback на fixed-size.

### 10.3. Результат chunking

Обе стратегии должны формировать список `DocumentChunk`.

Chunking должен быть реализован так, чтобы можно было:

- запустить индексирование только для одной стратегии;
- запустить индексирование для обеих стратегий;
- сравнить результаты двух стратегий на одном и том же наборе документов.

## 11. Метаданные чанков

Каждый чанк должен содержать минимум:

- `chunkId`: уникальный id;
- `documentId`;
- `strategy`: `fixed` или `structured`;
- `sourceType`: `readme`, `markdown`, `pdf`, `code`, `text`;
- `filePath`;
- `title`;
- `section`;
- `text`;
- `startOffset`;
- `endOffset`.

Желательно дополнительно:

- `language` для кода;
- `tokensApprox`;
- `createdAt`.

## 12. Генерация embeddings

Система должна:

- отправлять текст чанка в локальный Ollama;
- использовать embedding-модель, задаваемую в конфиге;
- сохранять embedding как массив `Float`.

Базовая модель:

- `nomic-embed-text`

Базовый endpoint:

- `http://localhost:11434/api/embed`

Требования:

- предусмотреть retry на временные ошибки;
- логировать неудачные чанки;
- не валить всю индексацию из-за одной ошибки;
- по возможности поддержать batching, но это не обязательное требование MVP.

## 13. Сохранение индекса

Минимально необходимо сохранить:

- текст чанка;
- metadata;
- embedding.

Основной вариант хранения:

- SQLite.

Рекомендуемая схема:

- таблица `documents`;
- таблица `chunks`;
- таблица `embeddings`.

Допустимый дополнительный экспорт:

- `chunks.json`;
- `embeddings.json`;
- `comparison.json`.

Важно:

- SQLite является основным локальным индексом в первой версии;
- в MVP допустимо не реализовывать JSON-экспорт, если SQLite покрывает сценарии хранения и демонстрации;
- embeddings могут храниться как JSON-строка или в ином локальном формате, если решение остаётся понятным и воспроизводимым;
- схема должна быть спроектирована так, чтобы потом можно было связать SQLite-записи с внешним FAISS-индексом через `chunkId`.

## 14. Будущее развитие: поиск

Поиск не является обязательной частью первой версии, но должен быть предусмотрен архитектурно.

В следующей итерации система может поддерживать простой локальный поиск:

- взять текст запроса;
- получить embedding запроса через Ollama;
- посчитать cosine similarity по всем чанкам выбранной стратегии;
- вернуть `topK` наиболее релевантных чанков.

Результат поиска должен содержать:

- `chunkId`;
- `score`;
- `filePath`;
- `title`;
- `section`;
- фрагмент текста.

Первая версия будущего поиска:

- brute-force по всем чанкам;
- cosine similarity в приложении;
- сортировка по score.

Путь развития:

- выделить отдельную абстракцию поискового движка;
- текущая реализация: `BruteForceSearchEngine`;
- будущая реализация: `FaissSearchEngine`.

## 15. Сравнение стратегий chunking

Нужно реализовать сценарий сравнения `fixed` и `structured`.

Минимум сравнить:

- количество чанков;
- среднюю длину чанка;
- минимальную и максимальную длину;
- распределение размеров;
- качество retrieval на 3–5 тестовых запросах.

Оценка retrieval в учебной версии может быть ручной:

- смотреть, насколько релевантны результаты `top3` или `top5`;
- зафиксировать наблюдения в отчёте;
- сделать короткий вывод, где fixed лучше, а где structured лучше.

Ожидаемый артефакт:

- `comparison.md` или `comparison.json`.

Примеры запросов:

- как запускается проект;
- где описана конфигурация;
- что делает конкретный сервис;
- как работает индексация pdf;
- где описаны зависимости.

## 18. CLI-сценарии

Проект должен поддерживать команды вида:

```text
index --input ./docs --strategy fixed
index --input ./docs --strategy structured
index --input ./docs --all-strategies
search --query "как запускается проект" --strategy fixed --top 5
compare --input ./docs
```

Кроме запуска через аргументы, проект должен поддерживать интерактивный CLI-режим.

Минимальное поведение интерактивного режима:

- после запуска без аргументов приложение остаётся в пользовательской сессии;
- отображает prompt для ввода команды;
- поддерживает команды `help` и `exit`;
- принимает те же команды, что и обычный CLI-режим, но в интерактивном формате;
- используется как основной ручной режим проверки во время разработки.

Допустим и такой вариант:

```text
search --query "..." --index ./data/index.db
report --compare
```
