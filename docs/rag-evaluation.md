# Evaluation explainable RAG

Этот документ описывает текущий формат проверки для `day_25`.

Актуальное направление проекта:

- основной RAG-режим — `model-rerank`;
- подтверждённый ответ должен иметь источники;
- подтверждённый ответ должен иметь одну лучшую цитату;
- при слабом контексте система должна честно отказываться от ответа.

## Что делает скрипт

Сценарий:

- `scripts/run-rag-evaluation.ps1`

выполняет следующие шаги:

1. Индексирует корпус для выбранной chunking-стратегии.
2. Для каждого контрольного вопроса запускает:
   - `plain`
   - `rag + model-rerank`
3. Для RAG-ответа отдельно извлекает:
   - `answer body`
   - `sources block`
   - `quotes block`
   - `retrieval block`
4. Строит два markdown-отчёта:
   - summary report
   - raw report

## Что именно проверяется

### 1. Есть ли источники

Скрипт смотрит, есть ли в ответе непустой блок `Источники`.

Для этого он ищет структурированные поля вида:

- `source = ...`
- `section = ...`
- `chunkId = ...`

### 2. Есть ли цитата

Скрипт проверяет, есть ли в ответе блок `Цитаты` и содержит ли он хотя бы одну запись:

- `chunkId = ...`
- `quote = ...`

Текущее ожидание проекта: для подтверждённого grounded-ответа достаточно **одной** хорошей цитаты.

### 3. Есть ли смысловое пересечение ответа и цитаты

Это не строгая семантическая метрика, а учебная эвристика.

Скрипт:

1. берёт значимые термины из `answer`;
2. берёт значимые термины из текста цитаты;
3. считает их overlap.

Если overlap слишком слабый, ответ не должен считаться уверенно grounded.

### 4. Похоже ли это на refusal

Отдельно отмечается, похож ли ответ на честный отказ:

- `не знаю`
- `уточните вопрос`
- `недостаточно данных`
- `контекст не найден`

Это важно, потому что отказ должен распознаваться как нормальная защитная реакция системы, а не как обычный answer.

## Как устроен verdict

Используются три учебных verdict-а:

- `success`
- `partial`
- `fail`

Их смысл:

- `success` — ответ grounded, есть источники, есть цитата, цитата заметно поддерживает ответ;
- `partial` — есть полезный сигнал, но один из ключевых признаков слабый или отсутствует;
- `fail` — ответ плохо опирается на retrieval, не имеет структуры или уходит от контекста.

## Summary report

Файл:

- `data/rag-evaluation-summary.md`

Это короткий отчёт для быстрого чтения.

Для каждого вопроса он показывает:

- формулировку вопроса;
- expectation;
- expected sources;
- таблицу по режимам;
- краткий вывод по качеству ответа.

В текущем состоянии основное сравнение идёт между:

- `plain`
- `RAG Model Rerank`

## Raw report

Файл:

- `data/rag-evaluation-raw.md`

Нужен для отладки.

Для каждого вопроса и каждого режима там отдельно видны блоки:

- `Answer`
- `Sources`
- `Quotes`
- `Retrieval`

Это помогает понять, на каком именно шаге ломается groundedness:

- retrieval;
- структура ответа;
- наличие цитаты;
- совпадение ответа с цитатой.

## Как запускать

Обычный порядок:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start-manual-check.ps1
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\run-rag-evaluation.ps1
```

После этого полезно открыть:

- `data/rag-evaluation-summary.md`
- `data/rag-evaluation-raw.md`

## Operational note

Полный evaluation зависит от внешнего LLM-агента.

Если агент возвращает `403` или `agent_not_active`, evaluation нужно перезапускать только после активации этого агента.

Также должна быть доступна локальная embedding/Ollama-инфраструктура, иначе индексирование и retrieval не смогут отработать.

## Как интерпретировать результат

Для учебного проекта важно смотреть не только на то, “есть ли ответ”, но и на то, **можно ли ему доверять**.

Если:

- retrieval выглядит разумно;
- источники есть;
- цитата есть;
- цитата поддерживает ответ,

то это хороший grounded case.

Если ответ есть, но цитаты нет, это полезный, но ещё не полностью подтверждённый результат.

Если система честно отказывается отвечать на слабом контексте, это тоже положительное поведение, а не провал.
