# ТЗ: переход на `model-rerank` и упрощение prompt для цитат

## Контекст

В текущем `day_25` проект уже умеет:

- выполнять retrieval по embeddings;
- применять несколько post-retrieval режимов;
- формировать структурированный RAG-ответ;
- возвращать источники и цитаты;
- включать режим честного отказа при слишком слабом контексте.

По итогам обсуждения в этой сессии принято решение:

1. больше не развивать несколько post-retrieval стратегий параллельно;
2. использовать **только `model-rerank`** как основную и целевую стратегию отбора контекста;
3. пока **не усложнять guard на релевантность**;
4. отдельно доработать **prompt для цитат**, чтобы повысить стабильность grounded-ответов.

---

## Цель

Сфокусировать RAG-пайплайн на одном основном post-retrieval режиме и повысить вероятность того, что модель:

- возвращает полезный ответ;
- возвращает хотя бы одну цитату;
- делает это в более стабильном и простом формате.

---

## Что меняем

### 1. `model-rerank` становится основной стратегией

Нужно привести проект к такому состоянию, где `model-rerank` рассматривается как основной и рекомендуемый post-mode.

Это означает:

- в конфиге по умолчанию для RAG должен использоваться `model-rerank`;
- в README и документации `model-rerank` должен описываться как основной режим;
- ручные сценарии проверки должны использовать `model-rerank` как базовый режим;
- evaluation и примеры использования должны в первую очередь опираться на него.

Важно:

- режимы `none`, `threshold-filter`, `heuristic-filter`, `heuristic-rerank` не обязательно немедленно удалять из кода;
- но они перестают быть основным направлением развития;
- дальнейшие улучшения качества контекста нужно в первую очередь делать внутри логики `model-rerank`.

### 2. Улучшаем prompt для цитат

Сейчас одна из основных проблем — модель часто возвращает ответ без цитат или с пустым `quotes`.

Новый prompt должен сделать задачу проще и жёстче:

- модель должна вернуть **ровно одну лучшую цитату** вместо диапазона `1..3`;
- цитата должна быть коротким дословным фрагментом из переданного контекста;
- `chunkId` должен быть одним из реально переданных в prompt;
- если модель не может подтвердить ответ хотя бы одной цитатой, она не должна пытаться изображать уверенный grounded-ответ.

То есть мы уменьшаем сложность задачи для LLM:

- не несколько цитат;
- не “цитаты по желанию”;
- а одна лучшая цитата как обязательный минимум для подтверждённого ответа.

---

## Функциональные требования

### A. Изменение default-режима

В проекте нужно обновить default post-mode для RAG на `model-rerank`.

Ожидаемое поведение:

- если пользователь запускает `ask --mode rag` без явного `--post-mode`,
- то используется `model-rerank`.

Проверить:

- конфиг;
- загрузку конфига;
- runtime-значения по умолчанию;
- документацию и примеры.

### B. Новый контракт prompt-а для цитат

Модель должна получать инструкцию следующего смысла:

1. отвечать только по найденному контексту;
2. вернуть JSON;
3. JSON должен содержать:
   - `answer`
   - `quotes`
4. `quotes` должен содержать **ровно 1 цитату** для подтверждённого ответа;
5. каждая цитата должна содержать:
   - `chunkId`
   - `quote`
6. `quote` должен быть дословным фрагментом из контекста;
7. если цитату дать нельзя, модель должна явно показать, что подтверждённый ответ не получается.

### C. Поведение при отсутствии цитаты

Текущую мягкую UX-логику можно сохранить:

- если ответ есть, но цитат нет, ответ не показывается как полностью подтверждённый;
- CLI сообщает, что ответ не удалось подтвердить цитатами.

Но цель доработки prompt-а — сократить число таких случаев.

### D. Evaluation и ручная проверка

После изменения prompt-а нужно повторно оценить:

- есть ли цитаты в ответах;
- стало ли больше успешных grounded-ответов;
- не ухудшилось ли качество самих ответов.

Основной режим сравнения:

- `rag + structured + model-rerank`

---

## Нефункциональные требования

### 1. Не делать большой рефакторинг retrieval guard

На этом этапе не нужно:

- вводить сложный semantic guard;
- добавлять новые composite relevance score;
- внедрять отдельную сущностную или lexical валидацию до ответа.

Эта часть сознательно откладывается.

### 2. Не усложнять контракт ответа сверх необходимого

Не нужно:

- возвращать несколько разных типов цитат;
- добавлять длинные reasoning-поля;
- строить многоступенчатую схему JSON.

Нужен простой и стабильный контракт.

### 3. Сохранять учебную прозрачность

Изменения должны оставаться понятными:

- по коду;
- по CLI;
- по документации.

## Критерии готовности

Задача считается выполненной, если:

1. `model-rerank` установлен как основной default post-mode;
2. prompt изменён так, что модель ориентируется на одну лучшую цитату;
3. проект успешно собирается;
4. ручной сценарий `rag + model-rerank` работает без дополнительных override;
5. документация отражает новое направление;
6. evaluation можно запускать без противоречий с новым форматом цитат.

---

## Что сознательно не делаем сейчас

На этом этапе **не входит в работу**:

- удаление старых post-mode из кода;
- сложный guard на слабую релевантность;
- расширенная валидация цитат;
- retry-механизм для повторного запроса цитаты;
- переход к нескольким подтверждающим цитатам.

---

## Рекомендуемый порядок реализации

1. Перевести default post-mode на `model-rerank`.
2. Обновить prompt в `RagQuestionAnsweringService`.
3. Проверить поведение CLI на ручных вопросах.
4. Обновить evaluation при необходимости.
5. Обновить README и docs.

---

## Короткий смысл решения

Вместо того чтобы улучшать сразу всё:

- не распыляемся на несколько retrieval-стратегий;
- делаем ставку на `model-rerank`;
- упрощаем задачу модели;
- добиваемся более стабильного grounded-ответа через одну обязательную цитату.
