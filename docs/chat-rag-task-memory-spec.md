# ТЗ: мини-чат с RAG, источниками и памятью задачи

## 1. Контекст

На текущем этапе проект уже умеет:

- выполнять explainable RAG по локальному индексу;
- использовать `model-rerank` как основной post-retrieval режим;
- отвечать только с опорой на найденный контекст;
- показывать источники;
- показывать цитату как подтверждение ответа;
- честно отказываться от ответа при слабом контексте.

Следующий этап развития проекта:

- перейти от single-turn `ask --mode rag` к mini-chat сценарию;
- хранить историю диалога;
- добавлять память задачи поверх истории;
- не терять цель диалога на длинной переписке;
- сохранять explainable RAG-подход, а не заменять его обычным chat completion.

---

## 2. Цель этапа

Реализовать mini-chat, который:

- хранит историю диалога;
- на каждом пользовательском ходе обновляет `TaskState`;
- на каждом содержательном вопросе выполняет retrieval через RAG;
- отвечает с учётом найденного контекста и памяти задачи;
- всегда возвращает источники;
- устойчиво держит цель и ограничения диалога в сценариях по 10–15 сообщений.

Главная учебная цель этапа:

- показать, как поверх обычного RAG добавляется task-aware dialogue layer;
- отделить полную историю диалога от рабочей памяти задачи;
- не перегружать retriever всей перепиской, а формировать из памяти и текущего вопроса search-friendly запрос.

---

## 3. Главный принцип этапа

На этом этапе приоритет у прозрачности и управляемости пайплайна, а не у максимальной “умности”.

Это означает:

- полная история диалога хранится как source of truth;
- рабочая память хранится отдельно как компактный `TaskState`;
- `TaskState` обновляется отдельным шагом перед ответом;
- retrieval работает не по всей истории, а по специально собранному `retrievalQuery`;
- существующий explainable RAG-слой сохраняется как ядро ответа.

Дополнительный принцип текущего учебного этапа:

- обратная совместимость не является обязательной целью;
- если ради более понятной архитектуры нужно изменить внутренние контракты или упростить старый слой, это допустимо;
- приоритет у детально прослеживаемой структуры доработок и понятной эволюции проекта;
- при этом существующий сценарий `ask --mode rag` желательно не ломать без отдельной причины, но это не жёсткое ограничение уровня production.

---

## 4. Что должно получиться в итоге

Проект должен уметь:

- запускать chat-сессию в CLI;
- принимать несколько пользовательских сообщений подряд;
- хранить историю сообщений внутри сессии;
- обновлять `TaskState` на каждом пользовательском ходе;
- использовать `TaskState` для формирования retrieval-запроса;
- выполнять RAG на каждом содержательном пользовательском запросе;
- возвращать ответ с источниками;
- при необходимости показывать текущее состояние памяти задачи;
- проходить 2 длинных сценария по 10–15 сообщений без потери цели.

Итоговый результат этапа:

- chat-режим поверх текущего RAG;
- отдельная модель памяти задачи;
- отдельный шаг обновления памяти;
- отдельный шаг сборки retrieval query;
- 2 длинных проверочных сценария;
- документация, объясняющая весь пайплайн.

---

## 5. Границы этапа

### Входит в текущий этап

- CLI mini-chat;
- хранение истории в рамках chat-сессии;
- `TaskState` как отдельная структура;
- LLM-based update `TaskState` с JSON-контрактом;
- retrieval query builder;
- ответы через текущий explainable RAG;
- вывод источников в каждом ответе;
- 2 длинных сценария ручной или полуавтоматической проверки.

### Не входит в обязательный минимум

- web UI;
- долговременная память между разными чат-сессиями;
- сложное хранение в `SQLite` для истории чата;
- agentic planning;
- tool-use внутри диалога;
- multi-user chat;
- semantic memory graph;
- автоматическое разрешение конфликтов памяти без участия модели.

## 6. Основные сущности

### 6.1. История диалога

История диалога хранится как полный журнал сообщений сессии.

Минимально сообщение должно содержать:

- `turnId`;
- `role`;
- `text`;
- `timestamp`.

Пример роли:

- `user`;
- `assistant`.

История нужна как:

- источник истины для всей переписки;
- база для отладки;
- источник последних сообщений для update-пайплайна памяти.

### 6.2. Память задачи

Память задачи хранится как отдельная структура `TaskState`.

Минимальные поля:

- `goal` — текущая основная цель диалога;
- `constraints` — ограничения, запреты и обязательные требования;
- `fixedTerms` — зафиксированные термины и определения;
- `knownFacts` — подтверждённые факты, уже сообщённые пользователем;
- `openQuestions` — нерешённые вопросы и развилки;
- `lastUserIntent` — нормализованное описание текущего пользовательского намерения.

Важно:

- `TaskState` не заменяет историю;
- `TaskState` — это компактная рабочая память;
- `TaskState` должен быть коротким, стабильным и пригодным для передачи в prompt.

`TaskState` должен рассматриваться как bounded-state, а не как бесконечно растущий JSON-словарь.

Минимальные правила bounded-state:

- `goal` хранит только одну текущую главную цель;
- `constraints` хранят только актуальные ограничения и дедуплицируются;
- `fixedTerms` хранят только явно зафиксированные термины;
- `knownFacts` и `openQuestions` имеют ограниченный размер;
- при конфликте между новым и старым значением система должна предпочитать новое явно подтверждённое пользователем значение;
- при невалидном update-результате система откатывается к предыдущему `TaskState`.

### 6.3. Chat session

Сессия объединяет:

- `sessionId`;
- `messages`;
- `taskState`;
- `createdAt`;
- `updatedAt`.

Для MVP сессия может жить только в памяти процесса.

## 7. Архитектурное решение

### 7.1. Общая идея

Новый chat-слой строится поверх уже существующего RAG-ядра, а не вместо него.

Текущий рабочий путь должен стать таким:

`user message -> task state update -> retrieval query build -> retrieval -> answer guard -> LLM answer -> sources -> save assistant message`

### 7.1.1. Где живёт lifecycle chat-сессии

Lifecycle chat-сессии должен жить в CLI-слое, а не в общем `WorkflowCommandHandler`.

То есть:

- `App.kt` или отдельный chat-loop управляет активной chat-сессией;
- chat-loop держит `ChatSessionCoordinator`;
- coordinator уже вызывает memory update, retrieval query builder и RAG core;
- служебные команды `:memory`, `:history`, `:exit` обрабатываются на уровне chat-loop/coordinator, а не как обычные `ask`-запросы.

Это решение лучше соответствует текущей структуре проекта, где обычный workflow живёт в модели “одна команда -> один `CommandResult`”.

### 7.2. Разделение слоёв

- chat history layer — хранит полную историю сообщений;
- task memory layer — хранит и обновляет `TaskState`;
- retrieval query builder — превращает текущий вопрос и память задачи в запрос для retriever;
- existing RAG layer — ищет контекст, запускает guard, строит grounded answer;
- CLI formatter — показывает ответ, источники и при необходимости память задачи.

### 7.2.1. Граница между chat-слоем и текущим RAG

Текущий `RagQuestionAnsweringService` уже связывает retrieval и answer generation через один `question`.

Для chat-режима это недостаточно, потому что появляются отдельные сущности:

- `userMessage`;
- `retrievalQuery`;
- `recentHistory`;
- `taskState`.

Поэтому на этом этапе предпочтительно:

- не размывать текущий сервис точечными параметрами;
- добавить отдельный request/adapter слой для grounded answer в chat-режиме;
- либо новый chat-aware coordinator, который переиспользует `RetrievalPipelineService` и текущий RAG core без жёсткого дублирования пайплайна.

Рекомендуемое направление:

- сохранить существующий single-turn `ask` как отдельный сценарий;
- собрать новый chat-пайплайн поверх него;
- если потребуется изменить внутренний контракт RAG-слоя, делать это явно и документированно.

### 7.3. Почему не передавать в retriever всю историю

Причины:

- история быстро становится шумной;
- короткие реплики плохо подходят как самостоятельные retrieval-запросы;
- retriever должен получать компактную и понятную формулировку;
- `TaskState` позволяет удерживать контекст без разрастания prompt.

## 8. Логика обновления `TaskState`

### 8.1. Когда обновлять

`TaskState` нужно обновлять на каждом пользовательском ходе.

Причины:

- проще и предсказуемее поведение;
- меньше риск, что новый запрет или новая цель будут пропущены;
- легче объяснять и отлаживать пайплайн;
- для диалогов длиной 10–15 сообщений стоимость такого update-подхода приемлема.

На этом этапе не нужно обновлять `TaskState` “раз в N сообщений”.

### 8.2. Что подаётся на update

В шаг обновления памяти подаются:

- предыдущий `TaskState`;
- последние релевантные сообщения диалога;
- новое сообщение пользователя.

Рекомендуемый минимум:

- предыдущий `TaskState`;
- последние 4–6 сообщений;
- текущее сообщение пользователя.

### 8.3. Как обновлять

Обновление `TaskState` выполняется отдельным LLM-запросом с жёсткой JSON-схемой.

Модель должна:

- сохранить актуальные элементы предыдущего состояния;
- добавить новые ограничения и факты;
- убрать устаревшие или отменённые ограничения;
- обновить `lastUserIntent`;
- не выдумывать факты, которых не было в истории.

Если модель возвращает невалидный JSON:

- использовать предыдущий `TaskState`;
- не ломать основной диалоговый ответ из-за сбоя memory update.

### 8.4. Правила интерпретации полей

Для extraction-step нужно считать, что:

- `goal` — главная цель пользователя;
- `constraints` — обязательные ограничения, запреты и требования;
- `fixedTerms` — явно зафиксированные термины и их значения;
- `knownFacts` — подтверждённые факты, которые пользователь уже сообщил;
- `openQuestions` — нерешённые вопросы и развилки;
- `lastUserIntent` — что пользователь хочет от ассистента прямо сейчас.

Типовые сигналы:

- `goal`: “нужно”, “хочу”, “цель”, “в итоге”, “сделай так, чтобы”;
- `constraints`: “только”, “нельзя”, “не нужно”, “без”, “обязательно”, “должно”;
- `fixedTerms`: “под … понимаем”, “назовём”, “будем считать”, “термин”;
- `knownFacts`: утверждения пользователя о текущем состоянии проекта или предметной области;
- `openQuestions`: выбор без решения, явная неопределённость, конфликтующие варианты;
- `lastUserIntent`: повеление или текущий вопрос в последнем сообщении.

Важно:

- это не regex-правила для production-кода;
- это интерпретационные сигналы для LLM prompt-а и пост-обработки.

### 8.5. Merge semantics для `TaskState`

Чтобы `TaskState` был тестируемым и стабильным, нужно явно зафиксировать правила обновления.

Минимальные правила:

- если новое пользовательское сообщение явно заменяет старое ограничение, новое ограничение вытесняет старое;
- если новое сообщение только добавляет ограничение, старые ограничения сохраняются;
- ограничения дедуплицируются по нормализованному смыслу, а не только по полному текстовому совпадению;
- `knownFacts` по умолчанию append-only, если пользователь явно не опроверг старый факт;
- `openQuestions` закрываются только тогда, когда пользователь или grounded-ответ действительно снимает развилку;
- `lastUserIntent` всегда переписывается по последнему пользовательскому сообщению;
- при конфликте между памятью и неявной интерпретацией истории приоритет имеет явная новая формулировка пользователя;
- при невалидном JSON, конфликтном merge или пустом обязательном результате используется предыдущий `TaskState`.

Дополнительно нужно ограничить размер памяти:

- `knownFacts` — не более фиксированного числа элементов;
- `openQuestions` — не более фиксированного числа элементов;
- `constraints` — только актуальные ограничения без исторического хвоста;
- `fixedTerms` — только термины, реально нужные для текущей темы.

## 9. Логика сборки `retrievalQuery`

### 9.1. Зачем нужен отдельный `retrievalQuery`

Последняя пользовательская реплика не всегда подходит для retrieval напрямую.

Проблемные случаи:

- короткие уточнения;
- местоименные вопросы;
- зависимые от прошлого контекста фразы;
- служебные реплики.

Поэтому нужно собирать search-friendly запрос отдельно.

### 9.2. Из чего строится `retrievalQuery`

`retrievalQuery` должен строиться из:

- текущего пользовательского сообщения;
- `goal`;
- релевантных `constraints`;
- релевантных `fixedTerms`;
- при необходимости `lastUserIntent`.

### 9.3. Принцип сборки

Если вопрос самодостаточный, `retrievalQuery` может быть почти равен пользовательскому тексту.

Если вопрос короткий и зависит от контекста, запрос должен явно восстанавливать тему.

Пример:

Пользователь пишет:

`А как тогда хранить историю?`

Вместо запроса:

`А как тогда хранить историю?`

Нужно собирать запрос вида:

`Для mini-chat с RAG и памятью задачи в CLI-приложении: как хранить историю диалога в первой версии без лишней сложности?`

### 9.4. Когда retrieval можно не усиливать

Если сообщение пользователя является чисто служебным:

- `да`;
- `окей`;
- `продолжай`;
- `давай дальше`;

то допустимо:

- либо использовать сокращённый retrieval query;
- либо использовать предыдущий актуальный intent;
- либо не делать новый retrieval, если команда не требует нового знания из базы.

Для первой версии допустимо начать с более простого правила:

- выполнять retrieval на каждом содержательном пользовательском сообщении;
- короткие служебные реплики обрабатывать отдельно.

### 9.5. Классы пользовательского ввода

Для chat-режима ввод пользователя нужно разделять минимум на 3 класса.

#### A. CLI-служебные команды

Примеры:

- `:exit`
- `:memory`
- `:history`

Поведение:

- не идут в LLM;
- не идут в retrieval;
- не обновляют `TaskState` как обычное пользовательское намерение;
- обрабатываются локально в chat-loop.

#### B. Содержательные пользовательские сообщения

Примеры:

- новый вопрос по предметной области;
- уточнение цели;
- добавление ограничения;
- фиксация термина;
- запрос продолжить обсуждение по теме.

Поведение:

- сохраняются в history;
- обновляют `TaskState`;
- запускают сборку `retrievalQuery`;
- запускают retrieval;
- приводят к обычному grounded answer или refusal.

#### C. Короткие служебные реплики внутри диалога

Примеры:

- `да`
- `окей`
- `продолжай`
- `давай дальше`

Поведение первой версии:

- сохраняются в history;
- обновляют `lastUserIntent`, если это уместно;
- по умолчанию не запускают полный retrieval по сырому тексту;
- используют либо предыдущий актуальный intent, либо отдельное правило `retrieval skipped`;
- их поведение должно быть явно зафиксировано в trace.

## 10. Логика генерации ответа

### 10.1. Порядок шагов

На каждом пользовательском ходе система должна делать следующее:

1. Сохранить сообщение пользователя в историю.
2. Обновить `TaskState`.
3. Построить `retrievalQuery`.
4. Выполнить retrieval по локальной базе.
5. Применить существующий answer guard.
6. Сгенерировать grounded answer на основе:
   - текущего вопроса;
   - `TaskState`;
   - recent history;
   - найденного контекста.
7. Вернуть ответ пользователю.
8. Сохранить ответ ассистента в историю.

### 10.2. Что должно попасть в answer prompt

В answer prompt нужно передавать:

- текущее сообщение пользователя;
- краткий `TaskState`;
- короткий recent history;
- найденный retrieval-контекст;
- жёсткое требование отвечать только по найденному контексту;
- требование возвращать источники через уже существующий RAG-контракт.

### 10.3. Что не нужно делать

Не нужно:

- подменять RAG обычным chat completion по истории;
- разрешать модели отвечать только по `TaskState` без retrieval;
- удалять guard на слабый контекст;
- разрешать ответ без источников.

### 10.4. Правило для источников

Формулировку “всегда возвращать источники” в этом этапе нужно понимать так:

- для grounded answer источники обязательны;
- для refusal при слабом контексте допустим пустой список источников;
- для CLI-служебных команд источники не требуются;
- для коротких service turns с `retrieval skipped` система должна явно показать статус хода, но не обязана печатать блок `Источники`.

## 11. Хранение истории

### 11.1. MVP-хранение

Для первой версии история должна храниться in-memory в рамках текущей CLI-сессии.

Причины:

- проще реализация;
- нет лишней storage-сложности;
- достаточно для сценариев из задания;
- удобно для учебного reference.

### 11.2. Расширяемость

При этом архитектура должна позволять позже заменить in-memory storage на файловое или database-based хранение.

Для этого желательно ввести абстракцию вроде:

- `ChatSessionStore`

с первой реализацией:

- `InMemoryChatSessionStore`

Опциональное будущее продолжение:

- `FileChatSessionStore` в `data/chat-sessions/*.json`

### 11.3. Почему не `SQLite` сразу

На этом этапе не нужно переносить chat history в `SQLite`, потому что:

- это даёт мало пользы для MVP;
- усложняет storage-слой;
- уводит фокус с основной идеи task-aware RAG.

## 12. CLI-сценарий

### 12.1. Минимальная новая команда

Нужно добавить точную MVP-команду:

```text
chat --strategy structured --top 3
```

Именно этот контракт нужно считать базовым для документации, help и ручной проверки.

Допустимые дополнительные параметры можно расширять позже, но первая версия должна опираться на одну конкретную форму запуска.

### 12.2. Минимальное поведение CLI

В chat-сценарии пользователь должен иметь возможность:

- начать новую сессию;
- последовательно отправлять сообщения;
- получать RAG-ответы с источниками;
- завершить сессию;
- при необходимости посмотреть текущее `TaskState`.

Допустимые служебные команды:

- `:exit`
- `:memory`
- `:history`

### 12.2.1. Старт и завершение chat-сессии

При запуске `chat --strategy structured --top 3` CLI должен:

- создать новую пустую chat-сессию;
- показать краткую стартовую подсказку;
- явно сообщить, что история живёт только в рамках текущего процесса;
- перечислить доступные служебные команды.

Пример ожидаемого старта:

```text
Mini-chat с RAG и памятью задачи.
История хранится только в текущей сессии.
Доступные команды: :memory, :history, :exit
Введите сообщение:
```

Команда `:exit` должна:

- корректно завершать сессию;
- не сохранять историю между запусками;
- завершать chat-loop без отправки данных в LLM.

Сигналы завершения процесса вроде `Ctrl+C` или EOF также желательно трактовать как корректное завершение сессии.

### 12.3. Требования к выводу

На каждый grounded ответ CLI должен показывать:

- ответ;
- источники;
- цитату, если она была получена текущим RAG-слоем;
- warning или refusal, если контекст слабый.

Для `:memory` CLI должен показывать:

- `goal`;
- `constraints`;
- `fixedTerms`;
- `knownFacts`;
- `openQuestions`;
- `lastUserIntent`.

Для `:history` CLI должен показывать:

- последние сообщения сессии в компактном формате;
- роль сообщения;
- при необходимости номер хода;
- ограниченный объём вывода по умолчанию.

Рекомендуемое поведение первой версии:

- `:history` показывает последние `N` сообщений;
- длинные сообщения обрезаются;
- служебные CLI-команды не смешиваются с обычными сообщениями как равноправные chat turns.

## 13. Предлагаемые изменения по слоям

### 13.1. CLI

Нужно расширить:

- `DefaultCliOutputFormatter`
- интерактивную оболочку в `App.kt`

Что добавить:

- вывод `TaskState` по служебной команде.

### 13.2. Workflow / orchestration

Нужно добавить отдельный orchestration-слой для чата.

Рекомендуемые сущности:

- `ChatCommand`
- `ChatResult`
- `ChatSessionCoordinator` или `ChatQuestionAnsweringService`

Задача слоя:

- управлять session lifecycle;
- обновлять память;
- вызывать retrieval;
- вызывать answer generation;
- синхронно обновлять историю.

### 13.3. Chat models

Нужно добавить модели:

- `ChatSession`
- `ChatMessageRecord`
- `TaskState`
- `FixedTerm`

При необходимости:

- `TaskStateUpdateRequest`
- `TaskStateUpdateResult`

### 13.4. Memory update service

Нужно добавить отдельный сервис обновления памяти:

- `TaskStateUpdateService`

Он должен:

- принимать предыдущий `TaskState`;
- принимать recent history;
- принимать новое сообщение пользователя;
- вызывать LLM;
- валидировать JSON;
- возвращать новый `TaskState`.

### 13.5. Retrieval query builder

Нужно добавить отдельный компонент:

- `RetrievalQueryBuilder`

Он должен:

- принимать пользовательское сообщение и `TaskState`;
- строить понятный для retriever текстовый запрос;
- не дублировать всю историю;
- быть максимально прозрачным и дебажимым.

### 13.6. Переиспользование текущего RAG

Нужно по максимуму переиспользовать существующий:

- `RetrievalPipelineService`
- `RagQuestionAnsweringService`

При необходимости допустимо:

- расширить `RagQuestionAnsweringService`, чтобы он принимал отдельно `retrievalQuery`, `recentHistory` и `taskState`;
- либо добавить новый chat-aware orchestration поверх него.

Предпочтение на этом этапе:

- не ломать существующий single-turn `ask`;
- build new chat layer поверх current RAG core.

Если для этого понадобится изменить внутренние контракты и отказаться от обратной совместимости на уровне внутренних API, это допустимо, потому что проект учебный и приоритет у ясной архитектуры.

## 14. Этапы реализации

### Этап 1. Введение chat-моделей

Цель:

- описать минимальные структуры для истории и памяти задачи.

Сделать:

- добавить `ChatSession`;
- добавить `ChatMessageRecord`;
- добавить `TaskState`;
- добавить модель `FixedTerm`.

Результат:

- у проекта появляется формальный контракт chat history и task memory.

### Этап 2. In-memory хранение chat-сессии

Цель:

- получить рабочее хранение истории без лишнего storage-слоя.

Сделать:

- добавить интерфейс `ChatSessionStore`;
- реализовать `InMemoryChatSessionStore`;
- поддержать создание и обновление chat-сессии.

Результат:

- chat-сессия может жить в рамках текущего процесса.

### Этап 3. Обновление `TaskState`

Цель:

- сделать отдельный шаг обновления памяти задачи.

Сделать:

- добавить `TaskStateUpdateService`;
- описать prompt для extraction;
- задать JSON-схему;
- добавить валидацию ответа модели;
- реализовать fallback на предыдущий `TaskState`.

Результат:

- система умеет извлекать и обновлять рабочую память задачи.

### Этап 4. Построение `retrievalQuery`

Цель:

- сделать retrieval более устойчивым к коротким и зависимым от контекста репликам.

Сделать:

- добавить `RetrievalQueryBuilder`;
- использовать текущее сообщение и `TaskState`;
- сделать правила для коротких служебных реплик;
- при необходимости логировать итоговый запрос в trace.

Результат:

- retriever получает search-friendly запрос вместо сырой реплики.

### Этап 5. Chat-aware orchestration

Цель:

- собрать новый полный chat-пайплайн.

Сделать:

- добавить chat coordinator;
- встроить шаги history -> task state -> retrievalQuery -> RAG -> save answer;
- переиспользовать текущий explainable RAG;
- не ломать существующий `ask`.

Результат:

- проект умеет отвечать в chat-режиме с памятью задачи.

### Этап 6. Расширение CLI

Цель:

- сделать новый chat-функционал доступным из консоли.

Сделать:

- добавить `chat` команду;
- добавить служебные команды `:exit`, `:memory`, `:history`;
- оформить человекочитаемый вывод ответов и memory snapshot.

Результат:

- chat-сценарий становится доступен для ручной проверки.

### Этап 7. Trace и debug visibility

Цель:

- сделать поведение memory-aware чата объяснимым и отлаживаемым.

Сделать:

- логировать `TaskState` update result;
- логировать предыдущий и новый `TaskState`;
- логировать итоговый `retrievalQuery`;
- логировать, был ли retrieval выполнен или пропущен;
- логировать причину skip-retrieval;
- логировать fallback на previous state при невалидном memory JSON;
- не перегружать обычный CLI debug-текстом.

Результат:

- длинные диалоги и ошибки памяти можно разбирать по trace.

Обязательный минимальный trace contract:

- `sessionId`
- `turnId`
- `requestId`
- raw user message
- previous `TaskState`
- new `TaskState`
- summary изменений в памяти
- final `retrievalQuery`
- retrieval action: `performed` или `skipped`
- skip reason, если retrieval пропущен
- guard outcome
- fallback reason, если memory update не применён

Рекомендуемые новые trace-события:

- `task_state_updated`
- `retrieval_query_built`
- `retrieval_skipped`
- `chat_turn_completed`

### Этап 8. Два длинных сценария проверки

Цель:

- проверить, что цель не теряется, а ответы остаются grounded.

Сделать:

- подготовить 2 сценария по 10–15 сообщений;
- прогнать их в chat-режиме;
- проверить удержание цели;
- проверить источники в каждом ответе;
- зафиксировать результаты в markdown-артефакте.

Результат:

- есть проверяемое подтверждение, что mini-chat работает по ТЗ.

Сценарии нужно делать не только narrative-описанием, но и turn-by-turn таблицами.

Для каждого хода нужно фиксировать:

- `input`
- ожидаемое изменение `TaskState`
- ожидаемое поведение retrieval
- ожидаемый invariant
- ожидаемый тип результата: grounded answer, refusal, service turn

## 15. Проверочные сценарии

### Сценарий 1. Уточнение цели и ограничений

В сценарии пользователь должен:

- сначала задать общую задачу;
- потом поэтапно уточнить ограничения;
- потом задавать короткие зависимые вопросы вроде “а как тогда лучше?”;
- затем проверить, что ассистент всё ещё отвечает в рамках исходной цели.

Проверить:

- цель остаётся стабильной;
- ограничения накапливаются корректно;
- retrieval не теряет тему;
- ответы содержат источники.

### Сценарий 2. Переопределение терминов и развилок

В сценарии пользователь должен:

- зафиксировать термины;
- уточнить несколько фактов;
- оставить часть решений открытыми;
- затем изменить одно важное ограничение;
- потом попросить продолжить уже с новым состоянием задачи.

Проверить:

- старое ограничение заменяется новым;
- открытые вопросы обновляются;
- `lastUserIntent` не застывает;
- ответы продолжают ссылаться на источники.

### Сценарии нужно оформить как turn tables

Для каждого сценария нужно сделать таблицу вида:

| Turn | User input | Expected TaskState delta | Retrieval | Expected result | Invariant |
| --- | --- | --- | --- | --- | --- |

Это нужно для того, чтобы критерии “цель не теряется” и “retrieval не теряет тему” были проверяемыми, а не оценивались только по общему впечатлению.

### Дополнительные негативные сценарии

Помимо 2 длинных сценариев, нужно добавить короткие acceptance-кейсы:

1. Слабый контекст и честный refusal.
2. Невалидный JSON от memory update и fallback на previous state.
3. Короткая служебная реплика после смены ограничения.
4. Поведение `:memory` и `:history` в начале и в середине сессии.

## 16. Критерии готовности

Этап считается завершённым, если:

1. В проекте есть отдельный chat-сценарий.
2. История диалога хранится в рамках chat-сессии.
3. `TaskState` обновляется на каждом пользовательском ходе.
4. `retrievalQuery` строится на основе текущего сообщения и `TaskState`.
5. Chat-ответы используют текущий explainable RAG-пайплайн.
6. В каждом grounded-ответе показываются источники.
7. Слабый контекст по-прежнему приводит к честному отказу.
8. Реализованы 2 длинных сценария проверки по 10–15 сообщений.
9. В длинных сценариях цель диалога не теряется.

### 16.1. Observable acceptance criteria

Каждый ключевой шаг должен иметь наблюдаемый результат.

Минимальный acceptance contract:

| Шаг | Вход | Ожидаемый observable result | Где проверять |
| --- | --- | --- | --- |
| `TaskState update` | user message + previous state | появляется новый state или зафиксирован fallback | trace / `:memory` |
| `retrievalQuery build` | user message + task state | виден итоговый search-friendly query | trace |
| `retrieval performed/skipped` | turn classification | фиксируется статус `performed` или `skipped` с причиной | trace |
| `guard refusal` | слабый контекст | assistant не выдумывает grounded answer | CLI output / trace |
| `grounded answer` | содержательный turn с контекстом | ответ содержит источники | CLI output |
| `service turn` | `:memory`, `:history`, `:exit` | ход не уходит в LLM | observable CLI behavior / trace |

### 16.2. Разделение проверок по слоям

Проверки должны быть разделены на 3 уровня.

#### Unit

Нужно покрыть без реальной модели:

- `ChatSessionStore`
- `RetrievalQueryBuilder`
- форматирование `:memory` и `:history`
- fallback-поведение при невалидном memory JSON
- trace emission
- классификацию service turns

#### Contract

Нужно покрыть:

- JSON-схему memory update;
- парсинг и валидацию результата;
- bounded semantics для `TaskState`.

#### Scenario

Нужно покрыть:

- 2 длинных диалога с реальной моделью;
- markdown transcript artifact;
- наличие refusal/sources там, где это ожидается.

## 17. Что сознательно не делаем сейчас

На этом этапе не входит в работу:

- хранение chat history между запусками приложения;
- полноценная long-term memory;
- semantic graph of facts;
- сложная классификация пользовательских реплик;
- многошаговый retry на memory update;
- перенос chat storage в `SQLite`;
- web-интерфейс;
- автоматическая внешняя оценка качества через отдельный judge LLM.

## 19. Короткий смысл решения

Мы не превращаем проект в “обычный чат с моделью”.

Мы добавляем поверх explainable RAG ещё один управляемый слой:

- полная история как журнал;
- `TaskState` как рабочая память;
- `retrievalQuery` как search-friendly формулировка;
- grounded answer как финальный ответ с источниками.

Именно эта комбинация должна дать mini-chat, который:

- помнит, о чём идёт речь;
- не теряет ограничения;
- продолжает отвечать по найденному контексту;
- остаётся понятным и проверяемым как учебный reference.

Отдельно важно:

- нам не нужна “production-строгость” в смысле обратной совместимости любой ценой;
- нам нужна прозрачная и детально прослеживаемая структура изменений;
- по итогам реализации должно быть легко увидеть, какие новые слои появились, как они взаимодействуют и как они меняют поведение проекта.
