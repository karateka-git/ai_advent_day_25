# ТЗ: RAG с цитатами, источниками и режимом "не знаю"

Ниже зафиксировано ТЗ в формате, с которым уже можно идти в реализацию. Текст разбит на блоки и поясняет не только что делать, но и зачем это нужно, чтобы проект оставался учебным и прозрачным.

## Цель

Доработать текущий RAG-режим так, чтобы каждый ответ в `ask --mode rag` возвращал:

- основной ответ на вопрос
- список источников
- цитаты из найденных чанков
- честный режим `не знаю`, если контекст недостаточно релевантен

Решение должно работать поверх уже существующих:

- post-retrieval режимов: `none`, `threshold-filter`, `heuristic-filter`, `heuristic-rerank`, `model-rerank`
- стратегий chunking: `fixed`, `structured`

## Идея Решения

В проекте уже есть хороший фундамент: любой `rag`-запрос проходит через retrieval pipeline и в конце даёт `selectedMatches`. Это и будет нашим ground truth для ответа.

Логика должна быть такой:

1. Сначала retrieval выбирает финальные чанки.
2. Затем система оценивает, достаточно ли этот контекст силён для ответа.
3. Если контекст слабый, ассистент не отвечает "по памяти", а говорит `не знаю` и просит уточнение.
4. Если контекст достаточный, LLM строит ответ по найденным чанкам.
5. Источники и цитаты выводятся как обязательная часть результата.

Важно: мы не переписываем retrieval-архитектуру. Мы усиливаем именно слой ответа поверх уже существующего retrieval.

## Функциональные Требования

### 1. Для каждого `rag`-ответа должна существовать единая структура результата

Структура должна содержать:

- `answer` — основной текст ответа
- `sources` — список источников
- `quotes` — список цитат
- `isRefusal` — флаг честного отказа
- `refusalReason` — причина отказа, если отказ был

Зачем это нужно:

Сейчас `RagAnswer` содержит только строку ответа и retrieval summary. Для учебного проекта это слишком неявно. Если сделать структуру явной, станет проще понимать, где именно лежит ответ, где источники, а где объяснимость.

### 2. Источники должны формироваться для каждого ответа

Для каждого использованного чанка надо выводить минимум:

- `source` — путь к документу или идентификатор источника
- `section` — секция документа
- `chunkId` — идентификатор чанка

Источник должен браться из `selectedMatches`, а не генерироваться свободно моделью.

Зачем это нужно:

Источники уже есть в retrieval. Это значит, что брать их лучше напрямую из данных поиска, а не просить модель придумывать их текстом. Так ответ становится более объяснимым и стабильным.

### 3. Цитаты должны присутствовать в каждом обычном RAG-ответе

Цитаты — это короткие фрагменты текста из найденных чанков, на которые опирается ответ.

Минимальные требования к цитатам:

- в каждом non-refusal ответе должна быть хотя бы 1 цитата
- желательно 1-3 цитаты
- каждая цитата должна быть привязана к `chunkId`

Зачем это нужно:

Источники говорят, откуда взят ответ. Цитаты показывают, на какой именно текст он опирается. Это самый наглядный анти-галлюцинационный слой для учебного RAG.

### 4. При слабом контексте система обязана отказаться от ответа

Если retrieval дал слабый результат, система должна вернуть примерно такой смысл:

- `не знаю`
- `уточните вопрос`
- `в найденном контексте недостаточно данных для уверенного ответа`

Этот режим должен включаться до генерации обычного ответа.

Зачем это нужно:

Это главный механизм anti-hallucination. Идея RAG не в том, чтобы "ответить любой ценой", а в том, чтобы ответить только тогда, когда найден опорный контекст.

## Нефункциональные Требования

### 1. Решение должно быть общим для всех `rag`-режимов

Не делать отдельную реализацию под каждый `post-mode`.

Почему:

Все режимы уже приводятся к одному формату через `RetrievalPipelineResult`. Значит, усиливать нужно общий слой ответа, а не плодить условные ветки.

### 2. Решение должно работать и для `fixed`, и для `structured`

Почему:

На слое ответа обе стратегии уже выглядят одинаково: есть текст чанка, `section`, `filePath`, `chunkId`.

### 3. CLI должен показывать новую структуру явно и читаемо

Пользователь должен видеть:

- ответ
- источники
- цитаты
- retrieval summary

Почему:

Для учебного проекта важна наблюдаемость. Нужно, чтобы можно было глазами понять, как система дошла до ответа.

## Предлагаемая Архитектура Изменений

### 1. Модель данных ответа

Файл: `src/main/kotlin/ru/compadre/indexer/qa/model/RagAnswer.kt`

Что добавить:

- новую структуру `RagSource`
- новую структуру `RagQuote`
- расширенный `RagAnswer`

Примерно логически:

- `RagSource(source, section, chunkId)`
- `RagQuote(chunkId, text)`
- `RagAnswer(answer, sources, quotes, isRefusal, refusalReason, retrievalResult)`

Почему:

Это превращает ответ из "просто текста" в полноценный объяснимый артефакт.

### 2. Логика определения слабого контекста

Файл: `src/main/kotlin/ru/compadre/indexer/qa/RagQuestionAnsweringService.kt`

Что добавить:

- проверку `selectedMatches.isEmpty()`
- проверку лучшего score против порога
- возможно, дополнительную проверку количества выбранных чанков

Если проверка не проходит:
возвращать refusal-ответ без обычной генерации.

Почему:

Если retrieval сам не нашёл хорошую опору, не надо заставлять LLM "спасать" ответ фантазией.

### 3. Конфиг порога для ответа

Файлы:

- `src/main/kotlin/ru/compadre/indexer/config/AppConfig.kt`
- `src/main/resources/application.conf`

Что добавить в конфиг:
отдельную секцию для answer-guard, например:

- минимальный score для уверенного ответа
- минимальное количество выбранных чанков
- возможно, флаг включения refusal guard

Почему:

Порог слабого контекста — это не бизнес-логика, зашитая в код. Это tuning-параметр, который полезно менять без переписывания реализации.

### 4. Новый prompt для RAG-ответа

Файл: `src/main/kotlin/ru/compadre/indexer/qa/RagQuestionAnsweringService.kt`

Что изменить:

- системный prompt
- user prompt с контекстом

Модель должна получать явную инструкцию:

- отвечать только по контексту
- обязательно вернуть ответ и цитаты
- не выдумывать факты вне контекста
- если данных недостаточно, говорить `не знаю`

Почему:

Даже при хорошем retrieval модель без явного формата часто начнёт отвечать "как удобно", а не "как требуется заданием".

### 5. Формирование источников

Файл: `src/main/kotlin/ru/compadre/indexer/qa/RagQuestionAnsweringService.kt`

Что сделать:

- после retrieval собрать `sources` из `selectedMatches`
- не делегировать это LLM

Почему:

Источники у нас уже известны и достоверны. Нет смысла просить модель генерировать то, что уже есть в системе.

### 6. Формирование цитат

Есть два варианта.

Вариант MVP:

- модель получает найденные чанки
- модель возвращает ответ и 1-3 цитаты

Почему это подходит сейчас:

Тяжёлая валидация цитат пока сознательно не включается. Для учебной версии этого достаточно.

Что важно:

- цитаты должны быть обязательным полем ответа
- они должны ссылаться на `chunkId`

### 7. Изменение CLI-вывода

Файлы:

- `src/main/kotlin/ru/compadre/indexer/workflow/result/CommandResult.kt`
- `src/main/kotlin/ru/compadre/indexer/cli/DefaultCliOutputFormatter.kt`

Что сделать:

Для `ask --mode rag` выводить отдельные блоки:

- `Ответ`
- `Источники`
- `Цитаты`
- `Retrieval-сводка`

Почему:

Так будет видно не только финальный текст, но и доказательную базу ответа.

### 8. Обновление workflow-слоя

Файл: `src/main/kotlin/ru/compadre/indexer/workflow/service/DefaultWorkflowCommandHandler.kt`

Что изменить:

- `AskResult` должен хранить не только `answer: String`, но и структурированный RAG-ответ
- для plain-режима оставить всё как есть
- для rag-режима передавать новую структуру в formatter

Почему:

Иначе CLI не сможет красиво и явно показать источники и цитаты как часть результата.

### 9. Обновление evaluation

Файлы:

- `scripts/run-rag-evaluation.ps1`
- `docs/rag-evaluation.md`
- возможно `README.md`

Что добавить в evaluation на 10 вопросах:

- есть ли в каждом RAG-ответе блок источников
- есть ли в каждом RAG-ответе блок цитат
- похож ли смысл ответа на смысл цитат
- срабатывает ли refusal там, где контекст слабый

Почему:

Сейчас evaluation проверяет в основном retrieval и overlap ответа с ожидаемыми фактами. По новому заданию нужно оценивать ещё и explainability.

## Как Проверять "Смысл Ответа Совпадает С Цитатами"

Это самый скользкий пункт, потому что автоматически понять "смысл совпадает" сложно.

Для MVP можно использовать простую эвристику:

- взять текст ответа
- взять все цитаты
- посчитать пересечение значимых терминов
- если ответ и цитаты почти не пересекаются, считать это проблемой
- спорные случаи оставлять на ручную проверку

Почему это нормально:

Для учебного проекта нам не нужна идеальная семантическая метрика. Нам нужен воспроизводимый сигнал, который помогает заметить проблему.

## Пошаговый План Реализации

1. Расширить модель `RagAnswer`.
2. Добавить конфиг порога слабого контекста.
3. Добавить guard в `RagQuestionAnsweringService`.
4. Изменить prompt так, чтобы модель возвращала ответ и цитаты.
5. Источники собирать из `selectedMatches`.
6. Протащить новую структуру через `AskResult`.
7. Обновить CLI formatter.
8. Обновить evaluation script.
9. Обновить документацию.
10. Прогнать 10 контрольных вопросов и зафиксировать результат.

## Что Здесь Самое Важное Для Понимания

Если совсем коротко, то в этом задании есть 3 ключевые идеи:

### 1. Retrieval и generation надо разделять

Retrieval отвечает на вопрос: "какие куски текста мы нашли?"
Generation отвечает на вопрос: "как по этим кускам сформулировать ответ?"

### 2. Источники и цитаты делают RAG объяснимым

Без них RAG легко превращается в "LLM что-то сказал, и мы надеемся, что это правда".
С ними можно проверить, откуда взялся ответ.

### 3. Честный отказ так же важен, как хороший ответ

Если система не умеет говорить `не знаю`, она почти неизбежно начнёт галлюцинировать.

## Границы MVP

Чтобы не раздувать учебный проект, я бы сознательно не включал сейчас:

- сложную пост-валидацию цитат
- повторные LLM-retry при плохом формате
- отдельный JSON parser с большой схемой и recovery-логикой
- тонкую статистическую оценку semantic faithfulness

Для первой версии достаточно:

- структурированного результата
- обязательных источников
- обязательных цитат
- refusal по порогу
- обновлённого evaluation

## Предлагаемый Порядок Работы

Я бы делал реализацию в таком порядке:

1. Сначала данные и контракт ответа.
2. Потом refusal guard.
3. Потом prompt и цитаты.
4. Потом CLI.
5. Потом evaluation.
6. В конце документация и ручная проверка.

Так проще отлаживать: мы сначала делаем каркас ответа, потом поведение, потом проверку.
