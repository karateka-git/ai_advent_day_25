# Development Policy

## Цель текущего этапа

На старте проект должен оставаться маленьким учебным reference, который прозрачно показывает:

- как запускается Kotlin CLI;
- как загружается конфиг;
- как разбираются команды;
- как постепенно наращивается индексирующий пайплайн.

## Технологический выбор

- основной стек проекта на текущем этапе: Kotlin;
- по умолчанию не распылять реализацию между несколькими языками, если Kotlin покрывает задачу;
- при выборе библиотек и структуры проекта предпочитать решения, которые естественно живут в JVM-мире.

## Предпочтения по архитектуре

- конфиг проекта и параметры запуска держать отдельно от прикладной логики;
- CLI parsing, orchestration и domain logic не смешивать в одном большом файле;
- важный orchestration-код по умолчанию документировать;
- новые абстракции добавлять только тогда, когда они реально упрощают следующий шаг реализации.

## Минимум без лишней сложности

- не добавлять в стартовый MVP лишние abstraction layers;
- сначала собирать рабочий end-to-end skeleton, потом расширять;
- если есть выбор между более быстрым и более сложным стартом, предпочитать решение, которое быстрее даёт рабочий результат без потери расширяемости.

## CLI и кодировка

- для русскоязычного CLI на Windows считать UTF-8 обязательной частью рабочего сценария;
- в клиентской точке входа явно настраивать `stdout` и `stderr` на UTF-8;
- если вывод в PowerShell выглядит некорректно, проверять и кодировку приложения, и кодировку shell-окружения.
- для ручной проверки предпочитать direct launcher из `build\install\...`, а не `gradlew run`, чтобы не смешивать вывод приложения с выводом Gradle.
- scripted-проверки и headless-сценарии держать в отдельных PowerShell-скриптах, а не зашивать в основной CLI.

## Future scope

- поиск и FAISS пока не реализуются;
- модели и структура проекта можно готовить к этим этапам, но не нужно внедрять преждевременную сложность.

## Зависимые операции

- зависимые шаги `build -> installDist -> run/manual-check` выполнять строго последовательно;
- не запускать параллельно несколько Gradle-команд, если они пишут в один и тот же `build/`-каталог;
- scripted-проверки, сборку и запуск launcher-а не совмещать в параллельных процессах без отдельной необходимости;
- при работе с Kotlin/Gradle помнить, что параллельный запуск зависимых шагов может ломать incremental cache и давать ложные ошибки сборки.
