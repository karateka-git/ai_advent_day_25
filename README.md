# local-document-indexer

Учебный Kotlin-проект, выросший из `day_24`.

Если в `day_24` основной фокус был на explainable RAG поверх `model-rerank`, то в текущем `day_25` поверх этого слоя добавлен **мини-чат с памятью задачи**:

- хранение истории диалога;
- retrieval на каждом содержательном ходе;
- ответы с источниками и цитатой;
- `TaskState`, который удерживает цель, ограничения, факты и текущее намерение пользователя.

## Что есть сейчас

Текущий chat/RAG-пайплайн выглядит так:

`user turn -> TaskState update -> retrieval query -> embedding retrieval -> model-rerank -> answer guard -> LLM answer/rewrite -> sources + quote`

В проекте уже есть:

- `chat`-режим в CLI;
- история сообщений внутри chat-сессии;
- `TaskState` с полями:
  - `goal`
  - `constraints`
  - `fixedTerms`
  - `knownFacts`
  - `openQuestions`
  - `lastUserIntent`
- unified memory-update шаг, который сразу определяет `turnType`:
  - `KNOWLEDGE_QUESTION`
  - `ANSWER_REWRITE`
  - `TASK_STATE_UPDATE`
  - `TOPIC_SWITCH`
  - `SERVICE_TURN`
- rewrite-path для ходов вроде `Коротко, в 2-3 предложениях.`;
- trace для chat-layer в:
  - `data/logs/rag-trace.jsonl`
  - `data/logs/rag-trace-readable.md`

## Ключевые изменения по сравнению с `day_24`

### 1. Добавлен mini-chat

Новая команда:

```text
chat --strategy structured --top 3
```

Внутри chat-сессии доступны:

```text
:memory
:history
:exit
```

### 2. Добавлена память задачи

Память обновляется на каждом пользовательском ходе через отдельный structured LLM-step.

Главная идея:

- хранить не всю историю в prompt целиком;
- выделять из истории компактное состояние задачи;
- использовать его для routing, retrieval и answer generation.

### 3. Retrieval стал intent-first

Сейчас retrieval-query строится в первую очередь из `lastUserIntent`, а не из длинного служебного prompt-подобного текста.

Это уже дало практический эффект:

- меньше query dilution;
- лучше проходят follow-up вопросы;
- меньше ложных refusal на коротких зависимых репликах.

### 4. Rewrite-ходы не запускают новый grounded answer

Если пользователь просит:

- сократить ответ;
- переформулировать;
- дать краткий итог;

то chat-слой переиспользует последний grounded answer вместо нового retrieval-answer цикла.

### 5. Источники и цитаты остаются обязательной частью ответа

В grounded-ответе CLI показывает:

- `Ответ`
- `Источники`
- `Цитаты`

Если ответ нельзя подтвердить найденным контекстом, система должна честно отказаться.

## Как запускать

### Сборка и запуск проекта

По проектному правилу:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start-manual-check.ps1
```

Этот скрипт:

- собирает проект;
- обновляет `installDist`;
- очищает `rag-trace`;
- поднимает direct launcher.

### Отдельный launcher

После сборки используется:

```text
build/install/ai_advent_day_25/bin/local-document-indexer.bat
```

### Построение индекса

Внутри CLI:

```text
index --input ./docs/articles/doroshevich --strategy structured
```

### Запуск чата

```text
chat --strategy structured --top 3
```

## Автопрогон в этой сессии

В этой сессии был выполнен автоматический прогон полного сценария удержания цели и ограничений для текста `Реформа`.

Прогоняемые реплики:

1. `Будем обсуждать только текст «Реформа». Отвечай только по нему.`
2. `Как описывается рождение богини Реформы?`
3. `Какие детали места и времени её рождения названы?`
4. `Как описана сама богиня Реформа?`
5. `Что говорится о её волосах и глазах?`
6. `Какие черты её образа особенно подчёркнуты?`
7. `Что происходит в её первый день?`
8. `Как она влияет на людей и происходящее вокруг?`
9. `Какие детали сцены показывают её силу?`
10. `Коротко, в 2-3 предложениях.`
11. `Каким итоговым образом показана богиня Реформа?`
12. `:memory`
13. `:history`
14. `:exit`

Сырые артефакты прогона:

- `build/tmp/eval/custom_reforma_scenario.out.txt`
- `build/tmp/eval/custom_reforma_scenario.err.txt`
- `data/logs/rag-trace.jsonl`
- `data/logs/rag-trace-readable.md`

### Что получилось

- сценарий в целом завершился успешно и CLI корректно обработал все 14 ходов;
- стартовая рамка ушла в `TASK_STATE_UPDATE_ONLY`, а не в ложный knowledge-answering;
- вопросы про рождение, место и время рождения, волосы и глаза, черты образа, влияние на людей, детали силы и итоговый образ дали grounded-ответы;
- `Коротко, в 2-3 предложениях.` корректно пошёл по `ANSWER_REWRITE_REUSE`;
- `:memory` и `:history` отработали без сбоев.

### Что не идеально

- вопрос `Что происходит в её первый день?` всё ещё ушёл в refusal, хотя соседние вопросы про ту же сцену позже уже отвечаются;
- `knownFacts` к концу диалога накапливает дублирующиеся или слишком близкие формулировки;
- `openQuestions` не всегда очищается после того, как ответ уже был получен;
- `:history` сейчас показывает хвост сессии, а не весь диалог целиком.

### Вывод по прогону

Полный сценарий показал, что текущий mini-chat уже умеет:

- удерживать цель диалога и ограничение `только текст «Реформа»`;
- обрабатывать несколько последовательных содержательных вопросов в одной теме;
- переиспользовать последний grounded-ответ для rewrite-ходов;
- сохранять и показывать `TaskState` и историю сессии.

Главный оставшийся узкий участок сейчас — retrieval/evidence для некоторых формулировок внутри одной и той же сцены:

- вопрос `Что происходит в её первый день?` ещё не получает достаточно сильный набор evidence;
- память задачи уже выглядит полезной, но требует более аккуратной дедупликации и закрытия `openQuestions`.

То есть следующий шаг улучшений — не общий рефакторинг chat-loop, а более точный подбор evidence для близких follow-up вопросов и более чистое обновление `TaskState` после успешных ответов.

## Что важно помнить

`day_25` — это уже не просто explainable single-turn RAG, а учебный production-like слой поверх него:

- chat с памятью;
- routing пользовательских ходов;
- retrieval на основе намерения;
- grounded answers с источниками;
- trace для разбора ошибок.

Главная цель проекта сейчас — не “получить любой ответ”, а удерживать цель диалога и давать **проверяемый ответ с опорой на контекст**.
