# local-document-indexer

Учебный Kotlin-проект, выросший из `day_23`.

Если в `day_23` основной фокус был на сравнении нескольких post-retrieval режимов, то в текущем `day_24` мы сузили направление и теперь развиваем **explainable RAG поверх одной основной стратегии — `model-rerank`**.

Главная цель текущего состояния проекта:

- отвечать только с опорой на найденный контекст;
- показывать источники;
- показывать цитату как подтверждение ответа;
- честно говорить `не знаю`, если контекст слишком слабый.

## Что сделано в этой сессии

### 1. Структурированный RAG-ответ

Ответ в режиме `rag` теперь строится как отдельная структура:

- `answer`
- `sources`
- `quotes`
- `warningMessage`
- `isRefusal`
- `refusalReason`

Это позволяет отделять:

- сам ответ;
- доказательную базу;
- честный отказ;
- предупреждение о том, что ответ не удалось подтвердить цитатой.

### 2. Источники стали обязательной частью ответа

Для `ask --mode rag` CLI показывает блок `Источники`, где для каждого источника выводятся:

- `source`
- `section`
- `chunkId`

Источники формируются из retrieval-результата, а не сочиняются моделью.

### 3. Цитирование упрощено до одной лучшей цитаты

Модель больше не пытается вернуть несколько цитат. Теперь prompt ориентирует её на **одну лучшую цитату**, которая:

- короткая;
- дословная;
- привязана к одному из переданных `chunkId`;
- напрямую поддерживает ответ.

Если цитата есть, она выводится в отдельном блоке `Цитаты`.

Если модель вернула ответ, но не дала подтверждающую цитату, CLI показывает предупреждение:

`Не удалось подтвердить этот ответ цитатой из найденного контекста.`

### 4. `model-rerank` стал основной стратегией

По итогам обсуждения мы перестали развивать несколько retrieval-стратегий параллельно и сделали ставку на `model-rerank`.

Сейчас именно он:

- используется как default post-mode в конфиге;
- считается основной рабочей стратегией для `rag`;
- лежит в основе ручной проверки и evaluation.

Другие режимы пока могут оставаться в коде как исторический слой, но текущее развитие проекта сосредоточено именно на `model-rerank`.

### 5. Guard на слабый контекст

Перед генерацией ответа работает guard, который проверяет:

- есть ли выбранные чанки;
- достаточно ли их;
- не слишком ли слабый лучший score.

Если контекст признан слабым, система отвечает в духе:

`Не знаю. Уточните вопрос: в найденном контексте недостаточно данных для уверенного ответа.`

Важно: этот guard пока ещё простой. Он хорошо ловит пустой или очень слабый retrieval, но ещё не идеально различает “высокий score, но плохой по смыслу контекст”.

### 6. Логирование проблемных ответов модели

Если модель:

- вернула невалидный JSON;
- не вернула обязательные поля;
- не смогла дать цитату;

сырой ответ пишется в:

- `data/logs/rag-llm-format.log`

Это помогает разбирать проблемы prompt-а и формата, не засоряя CLI.

### 7. CLI стал чище

В обычном `ask --mode rag` сейчас показываются:

- `Ответ`
- `Источники`
- `Цитаты` — только если они реально есть

Из повседневного CLI-вывода убраны:

- технические причины format failure;
- `Retrieval-сводка` в обычном `ask`.

## Что считаем основной архитектурой сейчас

Текущий рабочий путь выглядит так:

`query -> embedding retrieval -> model-rerank -> answer guard -> LLM answer -> sources + quote`

То есть:

- retrieval подбирает кандидатов;
- `model-rerank` выбирает лучший контекст;
- guard решает, можно ли вообще отвечать;
- LLM формирует ответ и одну лучшую цитату;
- CLI показывает ответ как explainable RAG-результат.

## Как запускать

### 1. Сборка и запуск CLI

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start-manual-check.ps1
```

### 2. Построить индекс

```text
index --input ./docs/articles/doroshevich --strategy structured
```

### 3. Ручная проверка основных сценариев

#### Есть ответ и цитата

```text
ask --query "Как в тексте «Реформа» описывается рождение богини Реформы?" --mode rag --strategy structured --top 3
```

Ожидается:

- есть `Ответ`
- есть `Источники`
- есть `Цитаты`

#### Есть ответ, но цитаты нет

```text
ask --query "Как в тексте «Аден» описываются основные цвета города и берега?" --mode rag --strategy structured --top 3
```

Ожидается:

- есть `Ответ`
- есть `Источники`
- блока `Цитаты` нет
- есть предупреждение о неподтверждённом ответе

#### Контекст слишком слабый

```text
ask --query "Как в тексте «Босфор» объясняется происхождение Босфора?" --mode rag --strategy structured --top 3 --post-mode heuristic-filter
```

Ожидается:

- после фильтрации не остаётся пригодного контекста;
- срабатывает guard;
- система отвечает `не знаю`.

Этот кейс всё ещё полезен как демонстрация честного отказа на пустом контексте, даже несмотря на то, что основная стратегия проекта теперь `model-rerank`.

## Evaluation

Полный сценарий проверки:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\run-rag-evaluation.ps1
```

Теперь evaluation ориентирован именно на:

- `plain`
- `rag + model-rerank`

Скрипт проверяет:

- есть ли источники;
- есть ли цитата;
- есть ли overlap между ответом и цитатой;
- выглядит ли отказ разумным.

Основные артефакты:

- `data/rag-evaluation-summary.md`
- `data/rag-evaluation-raw.md`
- `data/ten-question-citation-check.md`

## Что важно помнить

Текущий `day_24` — это не проект про “получить какой угодно ответ”, а проект про **проверяемый ответ**.

Главные идеи:

- retrieval и generation нужно разделять;
- `model-rerank` — основной способ выбрать лучший контекст;
- одна хорошая цитата полезнее, чем несколько нестабильных;
- честный отказ лучше, чем красивый, но неподтверждённый ответ.
